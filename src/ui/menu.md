功能模块与程序模块的 key 全局唯一，id 全局唯一。
即功能模块与程序模块的 key 不能相同。

id 被使用之后，就不能再修改

所以，页面标识与菜单标识要分开。菜单跟页面不是一对一关系，而是多对一关系。
所以，支持，在多个菜单打开同一个页面

## 菜单配置

在此处组合功能模块与程序模块，形成菜单。支持多级嵌套。

1. key1
   1. key11
2. key2
   1. key21
      1. key 211

## 说明

1. id 递增，程序模块的 id 中不要包含分组的信息，这样就可以随意分组与排序
2. 分组：{key,global},{title},{icon},{id},{path}
3. 页面: {key,global},{title},{icon},{id},[{path}]({ui_path})
4. 写一个工具，获取下一个可用的 id
5. id 确定后，不可再改变
6. id 从 100,000 开始
7. 增加一个 menu-lock.md 用于锁定分组与菜单的数字id，不允许用户编辑


menu_id   需填写
menu_name 需填写
parent_id 树状结构中已体现，无需填写
order_num 按照摆放的先后顺序，无需填写
path 需填写
component 是根据 {ui_key} 推导出来的 无需填写
is_frame  如果 path 以 `http(s)://` 开头，就认定为外链  无需填写
menu_type 不填写，有 {ui_key} 的就是菜单，否则为目录，按钮不在此处定义
visible 默认都不隐藏，需要隐藏的，专门一个区域定义 无需填写
status 都正常，无需填写
perms 默认都是 list，无需填写
icon 默认为 # 无需填写


注意，权限分为 API 与 UI

## 变更

1. 未执行的变更，要对key做有效校验
2. 已执行的变更，记录当时的内容和生成的文件列表，不再做有效校验，因为可能会修改
3. 在变更段落的侧边栏加一个运行按钮，支持批量操作
4. 变更执行过之后
   1. 在操作后面加上 git commit hash、操作时间
   2. 另起一行，子序列
   3. 源码文件，文件清单，相对于项目根目录
